### 1. No Email Notifications on Any Deal Transition (Critical for a trust product)

**Files:** `server/lib/email.ts`, `server/routes/deals.ts`

`email.ts` only has one function: `sendMagicLinkEmail()`. Nothing else. Zero emails are sent when:

- A deal is created (seller doesn't know they've been named in a deal)
- Payment is funded (seller doesn't know to ship)
- Item is marked shipped (buyer doesn't know to watch for delivery)
- Item is delivered (buyer doesn't know funds will auto-release in 48h)
- A dispute is raised (the other party and admin don't know)
- Auto-release fires (neither party is told funds moved)

The dispute modal UI says *"Both parties will be contacted by email"* — this is a lie. No email is sent. The ship modal says *"The buyer can now monitor delivery"* — also not true unless they happen to check the app.

This is the single most important missing feature. For a financial product where strangers are trusting each other with money, silent state transitions are a serious trust problem.

**What to build — add to `email.ts`:**

```ts
sendDealCreatedEmail(buyerEmail, sellerEmail, deal)     // → both parties
sendDealFundedEmail(sellerEmail, deal)                   // → seller: "ship now"
sendDealShippedEmail(buyerEmail, deal, trackingNumber)  // → buyer: "your item is on the way"
sendDealDeliveredEmail(buyerEmail, deal)                // → buyer: "confirm or it auto-releases in 48h"
sendDisputeRaisedEmail(buyerEmail, sellerEmail, deal)   // → both + admin
sendDealCompleteEmail(sellerEmail, deal)                // → seller: "funds transferred"
sendDealCancelledEmail(buyerEmail, sellerEmail, deal)   // → both parties
```

Call each one inside the relevant route handler after the `transitionDeal()` call. Wrap each in try/catch and log failures — a failed email should never prevent a deal transition from completing. The Resend integration is already in place; this is just adding more template functions and calling them.

### 2. SESSION_SECRET Warns But Does Not Hard-Exit in Production

**File:** `artifacts/api-server/src/app.ts` lines 70–74

```ts
const sessionSecret = process.env.SESSION_SECRET;
if (!sessionSecret) {
  logger.warn("SESSION_SECRET not set — using insecure dev fallback. Set this in production.");
}
```

A warning is not enough. If `SESSION_SECRET` is missing in production, sessions are signed with `"dev-secret-change-in-production"` — a public string that anyone can use to forge session cookies. The `checkStartupRequirements()` function in `startup-check.ts` correctly hard-exits on missing config, but `SESSION_SECRET` is not in that list.

**Fix — replace the warn with a throw:**
```ts
const sessionSecret = process.env.SESSION_SECRET;
if (!sessionSecret) {
  // Hard exit — missing secret means all sessions can be forged
  logger.error("FATAL: SESSION_SECRET environment variable is not set. Refusing to start.");
  process.exit(1);
}
```

Or add it to `checkStartupRequirements()` alongside `STRIPE_WEBHOOK_SECRET`.

### 3. No Deal Acceptance Step — Seller Never Agrees to the Deal Terms

**Files:** `server/routes/deals.ts`, `lib/db/src/schema/deals.ts`

The deal state machine still goes directly from `created` → `funded`. The seller is never asked to accept the deal before the buyer pays. This means:

- Seller has no chance to reject incorrect deal terms
- There is no record that the seller agreed to ship the described item at the agreed price  
- If a dispute arises, the seller can claim they never agreed to the terms as written

**What to build:**

Add `"pending_seller_acceptance"` as the initial state before `"created"`:

```ts
// Schema: add to dealStateEnum
"pending_seller_acceptance",

// State machine: new legal transitions  
pending_seller_acceptance: ["created", "cancelled"],  // seller accepts → created; seller rejects → cancelled
created: ["funded", "cancelled"],
```

When a deal is created by the buyer:
1. Deal goes to `pending_seller_acceptance`
2. Email sent to seller with a link: *"[Buyer] wants to buy [Item] for $X. Review and accept this deal."*
3. Seller clicks the link, signs in via magic link, reviews the deal, clicks Accept
4. Deal transitions to `created` — buyer is now prompted to pay
5. If seller doesn't respond in 72 hours, deal auto-cancels

This is a one-endpoint addition (`POST /deals/:dealId/accept`) plus the email template and a frontend acceptance page.

### 5. Dispute Form Has No Evidence Upload

**File:** `artifacts/safesend/src/pages/deals/detail.tsx` — `DisputeModal`

The dispute category and reason text are a good improvement. But the modal still says *"The admin will review evidence"* with no way to actually submit evidence. When an admin receives a dispute notification (once email is added), they'll have only a text description to work with.

**What to build:**

Add an optional file upload field to the dispute modal:
```tsx
<FormField name="evidence" render={({ field }) => (
  <FormItem>
    <FormLabel>Supporting evidence (optional)</FormLabel>
    <FormDescription>
      Photos, screenshots, or documents that support your case. Max 3 files, 5MB each.
```

### 6. No Scheduled Tracking Sync Job — Webhook Outage = Stuck Deals

**File:** `artifacts/api-server/src/index.ts`

The two scheduled jobs (`verifyShipments` every 6h, `autoReleaseDelivered` every 1h) are present and correct. But there is no scheduled job that polls TrackingMore for updates on active shipments.

Currently tracking updates only arrive via:
1. TrackingMore webhook push (primary path)
2. `syncTrackingFromTrackingMore()` called when a user loads the tracking page (on-demand only)

If TrackingMore has an outage or their webhook delivery fails, deals sit in `shipped` forever. The 48h flag catches the case where no scan arrives, but it doesn't advance deals to `delivered` — only a webhook or a user loading the page can do that.

**Fix — add a polling job:**
```ts
// In index.ts — run every 2 hours
setInterval(() => {
  pollActiveShipments().catch((err) => logger.error({ err }, "pollActiveShipments failed"));
}, 2 * 60 * 60 * 1000);
```

```ts
// New job in jobs/poll-shipments.ts
export async function pollActiveShipments(): Promise<void> {
  const active = await db.select().from(dealsTable)
    .where(eq(dealsTable.state, "shipped"));
  
  for (const deal of active) {
    if (deal.trackingNumber && deal.courierSlug) {
      await syncTrackingFromTrackingMore(deal.id, deal.trackingNumber, deal.courierSlug);
    }
  }
}
```

`syncTrackingFromTrackingMore()` already exists in `routes/tracking.ts` — extract it to a shared lib file so both the route and the job can call it.
